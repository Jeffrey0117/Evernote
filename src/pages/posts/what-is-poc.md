POC，Proof of Concept，概念驗證。

用最簡單的話說：**先做一個醜的、能動的東西，證明這條路走得通。**

## 一個經典案例：facebooc

GitHub 上有個專案叫 [facebooc](https://github.com/Bogdanp/facebooc)，作者是 Bogdanp。

故事是這樣的：他跟朋友打賭，說一個還行的開發者可以在幾天內用純 C 寫出一個基礎的 Facebook clone。為了增加難度，他只用標準 GNU C 函式庫和 SQLite3。

結果他真的做出來了。

README 裡有一行警告：「這只是一個有趣的練習。你不會想在生產環境中使用這些程式碼。」

這就是 POC 的精髓：**不是要做一個完美的產品，是要證明「這件事做得到」。**

## 什麼時候該做 POC？

**技術可行性不確定的時候。** 你不確定某個架構行不行、某個套件能不能滿足需求、某個演算法跑不跑得動。與其花三週做完才發現不行，不如花三天先驗證。

**要說服別人的時候。** 老闆問「這能做嗎？」，你說「能」，他可能半信半疑。但你丟一個能動的東西給他看，討論就不一樣了。

**學新技術的時候。** 想學一個新框架？別看三天文件，直接做一個小東西。能跑起來，你就懂了一半。

## 什麼時候不需要？

**技術棧很熟的時候。** 你已經用 React 做過十個專案了，再做一個類似的，不需要 POC。

**需求很明確的時候。** 客戶要的東西很清楚，技術方案也很標準，直接開工就好。

**時間不允許的時候。** 有時候就是要硬幹。但這種情況下，風險你要自己扛。

## POC 的重點

1. **快。** 幾天，最多一兩週。超過這個時間就不叫 POC 了。
2. **醜沒關係。** 程式碼亂、沒測試、UI 很醜，都可以。重點是驗證概念。
3. **可以丟掉。** POC 不是 v0.1，是草稿紙。驗證完就可以丟掉重寫。

facebooc 的作者很清楚這點。他沒有繼續把它做成真正的產品，因為那不是目的。目的是證明「用 C 寫 Facebook clone」這件事做得到。證明完了，結束。
